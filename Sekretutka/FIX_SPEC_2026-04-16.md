## 2. P0 — Срочный анблокер

### P0-1. SyntaxError на state 35 в двух нодах

**Корневая причина.**
В JSON-строке `jsCode` литералы промпта state 35 содержат `\n` вместо `\\n`. При JSON-парсинге `\n` становится реальным переводом строки, который попадает в одинарные кавычки JS-литерала → `SyntaxError: Invalid or unexpected token`. Остальные состояния корректно используют `\\n` (= `\n` в JS-исходнике, валидный escape).

**Где.**
1. `Onboarding FSM Router` → параметр `jsCode` → объект `prompts`, ключ `35:` (строка 99 в JSON-файле).
2. `Onboarding Draft Merge` → параметр `jsCode` → функция `promptFor` → объект `prompts`, ключ `35:` (строка 494 в JSON-файле).

**Побочный фикс.** По твоему утверждению `parse_mode=Markdown` (обычный, не MarkdownV2). Значит лишние `\\-`, `\\.`, `\\(`, `\\)`, `\\_` в тексте — мусор, который показывается как обратный слэш в Telegram. Убираю их в новой версии.

```js
      lines.push(`🩹 *Recovery (кастом):* ACTIVE — ${d.hp_rules.recovery_active_rule || '—'}; PASSIVE — ${d.hp_rules.recovery_passive_rule || '—'}.`);
    } else {
      lines.push(`🩹 *Recovery:* defaults из канона.`);
    }
  }

  if (Array.isArray(d.shop_items) && d.shop_items.length) {
    lines.push(`🛍 *Магазин (${d.shop_items.length}):* ${d.shop_items.map(s => `${s.name} — ${s.cost}`).join('; ')}.`);
  }

  if (Array.isArray(d.level_names) && d.level_names.length) {
    lines.push(`🏆 *Уровни:* ${d.level_names.join(' → ')}.`);
  }

  if (d.level_rules?.money_net_monthly_threshold != null && d.level_rules?.money_net_monthly_threshold !== '') {
    lines.push(`💰 *Money Net для апа:* ${d.level_rules.money_net_monthly_threshold} THB/месяц × 3 месяца.`);
  }

  lines.push('');
  lines.push('Если всё верно — напиши: *да, подтверждаю*.');
  lines.push('Чтобы поправить блок — напиши его название: био, реальность, спринт, задачи, привычки, KPI, цель, боссы, recovery, магазин, уровни, money net.');
  return lines.join('\\n');
}
```

### P1-4. Консистентность промптов привычек 17-22

**Проблема.**
- Промпт 17: `привычка — награда` (2 поля). `stateMeta[17]` ждёт `{habit:{name, reward_value}}`. Консистентно.
- Промпт 18: «К какому треку она относится?» → отдельный вопрос. `stateMeta[18]` ждёт `{track}`. Консистентно.
- Промпты 19, 20, 21, 22: «Пример: привычка — награда - трек» (3 поля в одном сообщении).
- НО `stateMeta[19]` ждёт `{habit:{name, reward_value}}` (2 поля), а трек спрашивается отдельным `stateMeta[20]`, `stateMeta[22]`.

**Последствие.** Пользователь пишет «медитация — 5 — vitality», код парсит только `парсить — 5` (parseDashFormat берёт первое и последнее число), трек теряется, система идёт в state 20 и спрашивает трек ещё раз.

**Решение (простое, хирургическое):** привести промпты 19-22 к формату state 17 — только «привычка — награда», без трека. Трек собирается следующим шагом (19 → 20 → 21 → 22 alternating: habit → track → habit → track). Это сохраняет текущую архитектуру 6 состояний (17-22) для 3 привычек.

*(Если делаем P2-1 «гибкие привычки 1-N», этот фикс становится неактуальным и его заменяет P2-1. Реши порядок в разделе 5, Open Q#3.)*

**Diff: `Onboarding FSM Router` и `Onboarding Draft Merge` → `prompts[19|20|21|22]`**

AFTER:
```js
19: 'Отлично. Давай вторую привычку в том же формате.\\n_Формат: привычка — награда_',
20: 'К какому треку относится эта привычка: 💪 Craft, 🫀 Vitality или 🤝 Social?',
21: 'И третья привычка.\\n_Формат: привычка — награда_',

```

## 7. Риски и открытые зоны неопределённости

**R1. CRM-верификация заблокирована.**
Google Drive MCP два раза упал по таймауту. Я не прочитала актуальный GAME_RULES на 2026-04-16. **Мой фолбэк:** в end-to-end тесте пункт 5 пройдёт вручную — ты посмотришь таблицу сама после деплоя и сравнишь с ожиданием. Можно переделать через экспорт GAME_RULES в CSV и загрузить в сессию.

**R2. Telegram Markdown render.**
Фикс state 35 переключает парсинг с MarkdownV2 на Markdown (обычный). Если в других состояниях использовались MarkdownV2 escape-символы — они покажутся как обратные слэши. Нужна ревизия всех 40 промптов на наличие `\\-`, `\\.`, `\\_`, `\\(`, `\\)`. Быстрый grep: `rg '\\\\[._\\-()]' "1.0 Sekretutka_Game Rules.json"`.

**R3. Дубли `buildContract` и `prompts`.**
Обе ноды (FSM Router и Draft Merge) содержат идентичные копии `prompts` и `buildContract`. Каждый фикс нужно применить ДВАЖДЫ. Пропущенный дубль = баг. В долгой перспективе лучше вынести в Set-ноду или отдельный sub-workflow, но это P4 рефакторинг.

**R4. `resource_type: 'XP\\\\_Gold'` (с двойным escape) в Draft Merge.**
При `task.resource_type = 'XP\\\\_Gold'` (4 backslash в JSON → 2 в JS → экранирование `_` для MarkdownV2). Это legacy MarkdownV2. После перехода на Markdown обычный — escape не нужен, в таблице осядет литерал `XP\\_Gold` вместо `XP_Gold`. **Не фикшу в этой спеке, чтобы не плодить изменения**, но помечаю в TODO для следующей итерации.

**R5. Неатомарный деплой.**
Обе ноды правятся отдельно в n8n UI. Между правкой первой и второй — система сломана. Стратегия: делать вне пиковых часов, сразу за двумя правками прогнать T-P0-1.
